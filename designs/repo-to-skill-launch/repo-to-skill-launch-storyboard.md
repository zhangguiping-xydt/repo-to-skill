# repo-to-skill Launch Storyboard

Format: 16:9, 1920×1080, approximately 70 seconds  
Route: Product narrative version  
Language: English primary, Chinese support copy

## Visual System

- Canvas: 16:9, 1920×1080
- Tone: calm, technical, trustworthy
- Motion: clean UI cards, terminal-inspired overlays, smooth left-to-right progress
- Color direction: deep charcoal background, soft cyan highlights, warm white copy
- Typography: bold English headlines, smaller Chinese support copy below
- Safety language: direct, visible, repeated only where useful

## Required Message Beats

1. **Opening**: repo-to-skill turns repository knowledge into an AI coding agent skill pack.
2. **Problem**: important project knowledge is scattered across files, scripts, and conventions.
3. **Local analysis**: the analysis is local-first and does not require uploading a repository.
4. **Skill generation**: output is a reviewable AI coding agent skill pack, not a black box.
5. **Validation and safety**: no upload, no remote DB, no default vector DB; validate/eval PASS before trust.
6. **Real CLI**: show that this is a practical command-line workflow.
7. **Closing**: position the product as a way to preserve and reuse project knowledge.

## On-screen Safety Badges

Use these badges across the middle and closing sections:

- Local-first

- No default vector DB
- Reviewable Skill
- validate PASS

## Suggested Terminal Overlay

Keep terminal text generic and safe for publication:

```bash
repo-to-skill analyze ./example-repo
repo-to-skill generate ./example-repo
repo-to-skill validate ./generated-skill
repo-to-skill eval ./generated-skill
```

Expected terminal result styling:

```text
analysis complete
generated reviewable skill
validate PASS
eval PASS
```

## Visual Notes by Section

### Opening

- Begin with a repository icon on a clean desktop-like canvas.
- The repository gently unfolds into layered knowledge cards.
- End the shot with the product promise: repository to AI coding agent skill pack.

### Problem

- Show the pain without blaming developers.
- Use copy that implies knowledge exists but is hard to transfer.
- Avoid cluttered or stressful visuals; keep it polished and product-focused.

### Local analysis

- Make the “local-first” idea visually obvious.
- Use a closed local loop animation around the repository.
- Do not show cloud upload arrows.

### Skill generation

- Show the generated skill as readable, editable content.
- Highlight “reviewable” as the trust mechanism.
- Avoid suggesting the system silently installs or applies unreviewed behavior.

### Validation and safety

- Use PASS badges only after validation and eval screens appear.
- Safety claims should be visible, not hidden in small legal-style text.
- Keep the message simple: local-first, no upload, no remote DB, no default vector DB.

### Real CLI

- Use generic commands and generic repository names.
- Do not display any internal, machine-specific, or private paths.
- Show the CLI as the bridge between repository analysis and practical use.

### Closing

- Bring the visual loop together: repo knowledge becomes assistant capability.
- End on product confidence, not hype.
- Leave the final frame readable for at least two seconds.
